**Decisions made where the prompt was ambiguous:** The master prompt's Section 3 `Constraints` shape (`rating?: number`, `occasion?: string[]`) does not exactly match the real frontend type in `extension/src/api/types.ts` (`rating?: { min: number }`, `occasion?: string`, singular). Decision: hardcode `http://localhost:8000` as a constant default in the new `src/api/client.ts` rather than building a settings UI (out of scope per Section 11's "do not add features beyond what's asked").

## 2026-07-22 — Backend implemented and smoke-tested end to end
**Master prompt section:** Sections 1, 2, 4, 5, 6, 7, 8 (nearly the whole backend)
**What was implemented:** Full FastAPI backend under `backend/app/`: SQLAlchemy models for `accounts`/`personas`/`profiles`/`events` (`models.py`), the event fold/projection logic (`projection.py`), the deterministic drift engine (`drift.py`), the Groq key-rotation client with strict/non-strict fallback (`ai/groq_client.py`), the two AI touchpoints (`ai/suggest_name.py`, `ai/drift_phrasing.py`), and all six endpoint groups (`routers/personas.py`, `routers/profiles.py`, `routers/ai.py`) wired together in `main.py` with CORS. Ran `docker compose up -d db` (Postgres 16-alpine on host port 5433) and `uvicorn app.main:app --port 8000 --reload`, then smoke-tested every endpoint with curl: persona create/list, profile create/list/get, drift (both a small in-place change and a big category change, confirming `update`/`new_profile` decisions), commit in all three modes, rollback, delete (soft), and `/ai/suggest-name`. The Groq calls succeeded on the first real key in rotation (key #2) using `strict: true` structured output with `openai/gpt-oss-20b` — both the AI-phrased drift reason and the AI-suggested name came back well-formed. CORS preflight against a `chrome-extension://` origin returns the correct `access-control-allow-origin` header.
**Decisions made where the prompt was ambiguous:** Found and fixed a real bug during smoke testing: `_next_version_number` (used for `mode: "new_version"` in the commit endpoint) assumed every `ProfileCreated`/`VersionCommitted` event payload has a `"version"` key, but `ProfileCreated` payloads don't carry one (version 1 is implicit) — this threw `KeyError: 'version'` on the very first `new_version` commit. Fixed by treating `ProfileCreated` as version 1 explicitly when computing the historical max. Verified the fix also confirms the earlier design decision (next version = historical max + 1, not projected-version + 1) actually works: created v1 → committed v2 → rolled back to v1 → committed new_version again → correctly got v3 (not a reused v2), then `update` correctly overwrote v3 in place, then `new_profile` correctly forked a new profile_id at v1.
Used SQLAlchemy's generic `JSON` column type for `events.payload` (not Postgres-specific `JSONB`) so the same model works unchanged if this ever needs to fall back to SQLite — no fallback was needed here since Docker was available, but this keeps the option open per Section 2 without extra code.
**Blockers hit / still open:** None. Backend is running locally at `http://localhost:8000` against Postgres on port 5433. Not yet deployed to AWS — CORS is currently permissive (`allow_origin_regex` for any `chrome-extension://*`), which is correct for local dev per Section 2 but must be tightened to the real extension ID before any public deployment.

## 2026-07-22 — Fixes from real-browser testing: missing `neck` field, incomplete URL round-trip, version labeling
**Master prompt section:** Section 3 (Constraints contract — extended, see decision below), Section 7 (drift weights), Section 5/6 (event payload + response shape for version labels/history)
**What was implemented:** The user actually loaded the unpacked extension in Chrome and tested it against live Myntra (something I hadn't been able to do myself — no browser-automation tool was available in my session). They found three real bugs:
1. **Only 3 of 6+ applied filters were captured.** `src/adapter/urlSchema.ts` only ever parsed `Brand`/`Fabric`/`Sleeve`/`Size`/`Color` out of Myntra's `f=` query param — it silently dropped `Neck` (no corresponding field existed anywhere in `Constraints`) and never parsed `Occasions` at all (despite `Constraints.occasion` already existing in the type). Fixed by: adding a new optional `neck?: { include: string[] }` field to `Constraints` (frontend `types.ts` + backend `schemas.py`, mirroring the existing `sleeve` field exactly), and parsing `Occasions` (Myntra's facet is comma-separated/multi-select, `Constraints.occasion` is a single string — joined multi-selections with `,` into one string rather than dropping all but one).
2. **Reopening a saved profile lost most of the original filters.** `buildUrlFromConstraints` only ever serialized `Brand`/`Fabric`/`Size`/`Color` back into the `f=` param — even fields that WERE captured (`sleeve`) were never written back out. Fixed by adding `Sleeve`, `Neck`, and `Occasions` to the URL builder too, so save → reload → reapply is now lossless for every field the parser captures.
3. **No way to name/label a version**, only ever "v1"/"v2". Extended the existing `commit` endpoint's already-present `name` field (previously only used for `mode=new_profile`) to also act as an optional version label for `new_version`/`update` — stored as `"label"` in the `VersionCommitted` event payload, propagated through `projection.py`'s fold, and surfaced two new additive response fields on `ProfileVersionOut`: `versionLabel` (the current version's label) and `history` (the full list of `{version, label, createdAt}` entries, deduped by version number so an `update`-mode commit correctly replaces its version's entry rather than duplicating it, per the "Update Current: overwrite silently" semantics already in the three-way save modal). No new request fields were needed anywhere.
**Decisions made where the prompt was ambiguous:** Adding `neck.include` as a new drift field would have broken the Section 7.2 invariant that weights sum to exactly 1.0. Rather than rebalancing every other field's weight (which the prompt explicitly says not to casually touch for `category.articleType`), I carved the new field's weight out of the existing `sleeve.include: 0.02` budget by splitting it evenly: `sleeve.include: 0.01` + `neck.include: 0.01`. This is the smallest possible change that preserves the sum-to-1.0 invariant and treats both as the same tier of "minor style attribute" they conceptually are.
**Blockers hit / still open:** Caught a real deployment bug while re-testing this: `uvicorn --reload` silently failed to actually restart the worker process after a file change (`WatchFiles detected changes... Reloading...` printed, but no new `Started server process` line ever followed) — the OLD code kept serving requests for several curl round-trips before I noticed the `neck` field was missing from responses despite being in the code. Killed the whole process tree and restarted without `--reload`; recommend NOT relying on `--reload` for the rest of this build — restart manually after each backend edit and verify via `/health` + a quick response-shape check, not just a 200 status, since a stale reload will still return 200s.

## 2026-07-22 — Second round of real-browser bugs: wrong facet key, and a generic catch-all for the rest
**Master prompt section:** Section 3 (Constraints contract, extended again)
**What was implemented:** More real-browser testing turned up that even the *named* fabric field wasn't working: Myntra's actual facet key is `"Fabric Types"`, not `"Fabric"` — the parser's `startsWith("Fabric:")` check never matched `"Fabric Types:Cotton"`, so every fabric filter was silently dropped despite the field existing. Rather than keep whack-a-moling one missing facet name at a time (this is the third one found this way, after `Neck` and now `Length`/`Fashion Trends`/etc. — Myntra exposes 20+ facets per category and hand-modeling every single one isn't worth it two days before the deadline), added a generic `Constraints.other: dict[str, list[str]]` catch-all field (both `schemas.py` and `types.ts`). The URL parser now extracts *every* `key:value` pair out of the `f=` param generically, pulls out the ones we explicitly model (brand, fabric — accepting both `"Fabric Types"` and `"Fabric"` — sleeve, neck, size, color, occasions), and stashes everything else verbatim in `other` so it round-trips losslessly through save → reload → reapply even though we don't compute drift on it.
**Decisions made where the prompt was ambiguous:** `other` is deliberately NOT in `drift.WEIGHTS` and never will be for arbitrary keys — it exists purely for display/round-trip fidelity, not scoring, since we can't assign a principled weight to a field we don't know the semantics of ahead of time. This is a scope line I'm drawing consciously given the remaining time before submission: named, weighted fields for what the drift engine should reason about; a generic bucket so nothing is ever silently lost even for facets we haven't gotten to.
**Blockers hit / still open:** None — verified against the exact URL from the user's screenshot (`Brand:LULU & SKY,StyleCast::Categories:Dresses::Color:Black_36454f::Fabric Types:Cotton::Length:Mini`) and confirmed `Fabric Types` and `Length` both parse correctly now (`Categories` is intentionally dropped as redundant with the pathname-derived category).

## 2026-07-22 — Root cause of "Save Profile does nothing, zero network requests"
**Master prompt section:** N/A — frontend store bug, not a spec section
**What was implemented:** The user reported clicking "Save Profile" produced no network request at all and the profile just never appeared. Traced it to `useShelfieStore.saveProfile`'s guard clause: `if (!state.liveConstraints || !state.activePersona) return;` — a bare silent `return` with no console output and no user-facing feedback. The screenshot showed the "add persona" input still open with a name typed but not yet confirmed (Enter never pressed), meaning `activePersona` was still `""` (no persona exists yet on a fresh account) when Save was clicked — the guard fired, `fetch` was never called, hence zero network activity and nothing appearing anywhere. This wasn't a backend bug at all.
**Decisions made where the prompt was ambiguous:** Rather than auto-creating a default persona (which would be a bigger, more opinionated UX change), added `console.error` diagnostics to every guard clause in the store (`saveProfile`, `requestSave`, `confirmSave`) so this class of silent no-op is immediately visible in devtools next time, and made `SaveSheet` itself refuse to render a functional-looking save form when there's no active persona — it now shows "Create or select a persona... before saving a profile" instead.
**Blockers hit / still open:** None.
